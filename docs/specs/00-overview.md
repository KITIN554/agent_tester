# 00. Обзор проекта и принципы реализации

## Что это за проект

Агент-тестировщик для двух экспериментальных агентских систем (Finance Agent и Travel Agent) — прототип методологии, разрабатываемой в магистерской ВКР «Разработка методологии тестирования и оценивания работы агентских систем».

## Связь с диссертацией

Глава 2 формализует методологию (метрики, рубрики, формат сценария, протоколы). Глава 3 — это реализация и применение. Все компоненты, которые здесь специфицируются, должны прямо реализовывать положения главы 2. Если в спецификации написано «согласно таблице 2.4», это значит, что код должен следовать именно этой таблице, а не «общепринятым практикам».

## Стартовый фундамент (УЖЕ ГОТОВО)

- `systems/finance_agent/` — рабочий QA-агент по личным финансам (одиночные задачи)
- `systems/travel_agent/` — рабочий многошаговый агент бронирования (диалоговые задачи)
- `data/` — синтетические данные (transactions.csv, destinations.csv)
- `.claude/agents/` — три sub-agent (scenario-generator, metric-analyzer, report-writer)
- `CLAUDE.md` — инструкции проекта

Эти компоненты НЕ перерабатываются. Новые компоненты должны их использовать как объекты тестирования и контекстный фон.

## Что предстоит реализовать

1. Агент-тестировщик (`tester/`) — оркестратор, исполнитель сценариев, судья, метрики, отчёты
2. Тестовые корзины (`baskets/`) — YAML-сценарии для обоих агентов
3. CLI (`tester` команда)
4. Локальный дашборд (`dashboard/`) — Streamlit
5. GitHub Actions для регрессии при push
6. Эволюционный генератор сценариев

## Порядок реализации

Спецификации пронумерованы — этот же порядок и есть порядок реализации. Каждая последующая спецификация может зависеть от предыдущих.01 — Архитектура агента-тестировщика (модули, интерфейсы)
02 — Формат YAML-сценария
03 — Метрики (RQS, PQS, RS, SS, ES)
04 — Судья (LLM-as-a-Judge / Agent-as-a-Judge)
05 — Gate-логика (правила допуска релиза)
06 — Reporter (HTML-отчёт)
07 — Dashboard (Streamlit)
08 — CLI
09 — GitHub Actions
10 — Эволюционный цикл (генератор сценариев)

Конкретные задачи для Claude Code лежат в `docs/specs/tasks/T<номер>-<имя>.md`.

## Принципы для Claude Code

### Стиль кода
- Python 3.11+, современные type hints (`list[str]`, `X | None`, `dict[str, int]`)
- Pydantic v2 для всех моделей данных
- async/await для LLM-вызовов
- `tenacity` для retry на каждом LLM-вызове (3 попытки, экспоненциальный backoff)
- `rich.console` вместо print
- ruff + mypy должны проходить чисто

### Тестирование
- Каждая публичная функция — pytest-тест в `tests/`
- Минимальная цель: тесты не падают (зелёные)
- Где есть LLM-вызовы — мокируй через `unittest.mock` или `pytest-mock`
- Цель покрытия: на старте 60%+, расти со временем

### Git-flow

**ВАЖНО:** Авто-push в main разрешён ТОЛЬКО при зелёных тестах.

Перед `git commit` обязательно:

```bashruff check .
ruff format --check .
pytest

Если хоть один шаг падает — НЕ коммить, доработай. Только зелёное идёт в main.

После коммита: `git push origin main`.

Сообщения коммитов — Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `test:`, `refactor:`.

### Что НЕ делать
- НЕ менять логику `systems/finance_agent/` и `systems/travel_agent/` — они тестируемые объекты
- НЕ хардкодить ключи API (используй `os.environ`)
- НЕ коммитить `.env`, `__pycache__`, `.venv`
- НЕ ломать совместимость с существующим `CLAUDE.md` и settings.json
- НЕ запускать команды, которые изменяют состояние GitHub-репозитория без явной задачи (force-push, удаление веток, создание/удаление репозиториев)
- НЕ модифицировать файлы «ядра» методологии: `tester/metrics.py`, `tester/gate.py`, `tester/judge.py`, `tester/models.py` без явной задачи в таске

### Что использовать
- Для LLM-вызовов: `OpenAI(api_key=os.environ["PROXY_API_KEY"], base_url=os.environ["PROXY_BASE_URL"])`, модель `os.environ["LLM_MODEL"]`
- Для CLI: `click`
- Для дашборда: `streamlit + plotly + pandas`
- Для шаблонов отчётов: `jinja2`
- Для YAML: `pyyaml` (`yaml.safe_load`)

## Процесс работы с задачами

Конкретные задачи лежат в `docs/specs/tasks/T<NNN>-<short-name>.md` и нумеруются последовательно. Каждая таска ссылается на одну или несколько спецификаций (00-10).

### Как давать задачу Claude Code локально

В Claude Code CLI (запущен в корне проекта):
реализуй задачу docs/specs/tasks/T002-models.md


Claude Code прочитает таск-файл, связанные спеки, реализует код, прогонит тесты, закоммитит и запушит в main (только при зелёных тестах).

### Как давать задачу через GitHub

Создай issue с текстом:@claude реализуй docs/specs/tasks/T002-models.md

Claude Code Action подхватит, реализует в feature-ветке, откроет PR. Ты ревьюнешь и сольёшь.

### Pre-commit правила

ВАЖНО для Claude Code: перед каждым `git commit` обязательно:

```bashruff check .
ruff format --check .
pytest

Все три должны быть зелёными. Если хоть один не проходит — **НЕ коммитить**, доработать. Это правило неприкосновенно.

### Порядок реализации тасков

Таски имеют номера и зависимости. Делать строго в порядке:
T001 → T002 → T003 → ... → T015.

Каждая таска в конце проверяет: тесты пройдены, push в main выполнен.

## Acceptance criteria для проекта в целом

После выполнения всех задач:

- [ ] `tester run --basket baskets/finance_agent` проходит без ошибок и создаёт отчёт
- [ ] `tester run --basket baskets/travel_agent` проходит без ошибок и создаёт отчёт
- [ ] `streamlit run dashboard/app.py` показывает результаты последних прогонов
- [ ] HTML-отчёты в `reports/runs/<run_id>/index.html` открываются в браузере
- [ ] GitHub Actions workflow запускается при push в main и пишет результат в комментарий PR (или artifact)
- [ ] Эволюционный sub-agent умеет сгенерировать новые сценарии для существующего агента, читая его код
- [ ] `pytest` зелёный
- [ ] `ruff check .` зелёный
- [ ] Все спецификации `docs/specs/*.md` реализованы
